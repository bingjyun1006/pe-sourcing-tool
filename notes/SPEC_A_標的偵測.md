## 一、模組定位

從外部文件與公開資源中，自動萃取符合條件的未上市 / 興櫃供應鏈隱形冠軍，沉澱至賽道樹狀圖。

### 鐵律
- **只碰 100% 事實**：嚴禁任何推論、猜測、關聯性連連看
- **Append-Only**：知識只增不改，保留完整歷史佐證鏈
- **本機優先**：所有資料存於本機，無雲端資料庫

---

## 二、目標提取範圍

| 類型 | 處理方式 |
|------|----------|
| ✅ 未上市公司（無任何股票代號） | 提取 |
| ✅ 興櫃公司（尚未正式掛牌，估值未被充分定價） | 提取 |
| ❌ 上市（TWSE）+ 上櫃（OTC）公司 | 排除 |

**排除判斷信號**：文件中該公司名稱旁附有四位數台股代號，或明確標示「上市」「上櫃」字樣者一律排除；興櫃代號或無代號者保留。

## 三、三條情報軌道

### 軌道總覽

| 軌道 | 方法 | 找到的目標類型 | 輸入方式 |
|------|------|--------------|---------|
| **軌道一：採購鏈穿透** | Layer 2/3 年報主要進貨商表 | 現貨型隱形冠軍（量產即戰力） | 分析師手動上傳 PDF |
| **軌道二：股權鏈穿透** | 財報附表九轉投資揭露 | 前瞻型黑科技新創（大廠押注） | 分析師手動上傳 PDF |
| **軌道三：外部資源爬取** | 政府補助公告 + 展會專區名冊 | 漏網之魚 + 技術交叉驗證 | AI 自動爬取 |

**Sweet Spot 邏輯**：同一家公司同時出現於多條軌道，信號強度倍增。
- 軌道一 + 軌道三（政府補助）→ 技術是真貨 + 已有營收
- 軌道二 + 軌道三（展會）→ 大廠在押注 + 市場開始關注
- 三軌同時出現 → 極高信號，優先立案

---

### 軌道一：採購鏈穿透

**核心邏輯**：大廠（Layer 1）年報供應商匿名，中型設備/材料廠（Layer 2/3）受「10% 進貨強制揭露條款」約束，必須實名揭露採購佔比達 10% 以上的供應商。因此跳過 Layer 1，直接讀 Layer 2/3 年報是關鍵破局點。

**輸入文件類型**：
- Layer 2/3 公司年報（主力：「主要進貨廠商表」章節）
- Layer 2/3 公司 CSR/ESG 報告（次要：案例小方塊去匿名化、衝突礦產熔煉廠名單）

**觸發方式**：分析師將 PDF 丟入 `raw_inputs/`，Watchdog 自動偵測。

**source_type**：`Annual_Report`、`CSR_Report`

---

### 軌道二：股權鏈穿透

**核心邏輯**：財務報告法規強制附表九「被投資公司相關資訊」，揭露母公司及其創投子公司的完整持股名單，包含完全未上市新創。持股 2–5% 的材料新創代表大廠超前部署風向；持股 >51% 代表核心技術控制。此軌道適用於大廠與中型廠，不受年報馬賽克影響。

**輸入文件類型**：
- 完整財務報告（Financial Statements）最後 5–10 頁附表九

**觸發方式**：分析師將 PDF 丟入 `raw_inputs/`，Watchdog 自動偵測。

**source_type**：`Financial_Statement`

**額外萃取欄位**：持股比例、原始投資成本、期末帳面價值（寫入 `context_relation`）

---

### 軌道三：外部資源爬取

**核心邏輯**：政府補助通過名單代表國家級技術審查背書；政府採購得標代表產品已驗證可商業化。兩者均為公開資源，由 AI 自動爬取，無需分析師手動操作。

**MVP 爬取來源**：

| 來源 | 機構 | 信號意義 | 可爬性 |
|------|------|---------|--------|
| TIIP 產創平台補助 | 經濟部產發署 | 技術研發獲國家審查認可，有明確技術方向，但可能仍在研發、尚未量產 | ✅ 完全可用 |
| SBIR 通過名單 | 中小企業署 | 早期小型研發公司，規模較小，技術方向有政策支持但成熟度待觀察 | ✅ SSL 可繞過（verify=False）|
| 政府採購網得標公告 | 公共工程委員會 | 產品已通過政府驗收交貨，三者中商業化程度最高，代表有真實客戶與營收 | ✅ 無需登入 |

三個來源合起來可呈現公司成熟度路徑：SBIR → TIIP → 採購網得標 = 從「早期研發」到「商業化落地」的完整軌跡。

**UI 呈現方式**：每個來源標題旁加 `ℹ` icon，hover 顯示上方信號意義說明（tooltip）。

**後續補充來源（Phase 2）**：

| 來源 | 說明 |
|------|------|
| TPCA Show 參展商 | 名單結構可爬，展前公布 |
| SEMICON Taiwan | 1,287 家參展商，需 AI 二次關鍵字篩選 |
| 104 職缺（驗證用）| 不做 discovery 主力，用於確認已知公司的技術方向與擴線狀況 |

**觸發方式**：分析師執行 `python librarian.py --new-track "賽道名稱"` 後，背景自動爬取並 Append 至研究簡報。

**source_type**：`Gov_Grant`、`Exhibition`

## 四、供應鏈層級地圖（Supply Chain Map）

### 設計原則
- **永遠是局部的、有佐證的**：地圖不宣稱完整，只顯示「已確認」vs「待發現」
- **兩段式呈現**：先骨架 → 隨上傳文件逐步填入
- **AI 推斷明確標注**：Layer 定位標「AI 推斷」，分析師可手動調整

---

### 狀態一：骨架地圖（輸入關鍵字後立即生成）

骨架地圖由 Gemini 生成，只填入**有把握的公開資訊**：

```
Layer 1│ 終端系統整合商 / 晶圓代工
       │ TSMC · NVIDIA（已知上市大廠，Gemini 填）

Layer 2│ 設備 / 測試 / 封裝服務
       │ 京元電子 · 帆宣 · 力成（已知上市，作為上傳入口推薦）

Layer 3│ 關鍵零組件 / 特殊材料
       │ ⬜ 待發現（上傳 Layer 2 年報後填入）

Layer 4│ 原料 / 前驅物 / 特殊氣體
       │ ⬜ 待發現
```

**骨架地圖限制**：
- Layer 1/2：僅列已知上市大廠（有把握的事實）
- Layer 3/4：一律留空，不猜，不用 AI 幻覺填入
- 每個 Layer 附一行功能描述（「這層在做什麼」）

**骨架地圖的用途**：告訴分析師「這個賽道的結構」→ 決定上傳哪間 Layer 2 的年報 → 開始挖 Layer 3

---

### 狀態二：填入地圖（上傳年報後自動更新）

上傳年報 → Gemini 發現 Layer 3 公司 → 自動插入地圖對應位置：

```
Layer 2│ 京元電子 ✅ 帆宣 ✅（已上傳年報）

Layer 3│ 漢辰科技 🟢（來源：京元電子年報 p.42）
       │ XX 供應商 🟡（來源：帆宣 CSR p.18）
       │ ⬜ 待發現

Layer 4│ ⬜ 待發現
```

**Layer 定位機制**：
- Gemini 從 context_relation 推斷放入哪一層（標注「AI 推斷」）
- 分析師可手動拖曳調整 Layer
- 分析師可手動新增公司（有自己的線索但不在系統資料裡）

**地圖的不完整性是功能，不是缺陷**：空格代表「這裡還有東西等待被發現」，是下一步研究的指引。

## 五、投資備忘錄（Company Fact Sheet）

**觸發條件**：分析師點擊某一發現的未上市公司，選擇「生成備忘錄」

**包含內容**（只保留高品質來源）：

| 章節 | 來源 | 品質 |
|------|------|------|
| 公司基本資料 | 商工登記（自動查詢）| ✅ 高 |
| 發現來源 | 三軌 evidence_quote 原文 | ✅ 高 |
| 客戶關係 | 年報採購揭露（10% 條款）| ✅ 高 |
| 股權結構線索 | 附表九持股比例 + 投資金額 | ✅ 高 |
| 政府補助驗證 | TIIP/SBIR 得標記錄 | ✅ 高 |
| 供應鏈定位 | AI 推斷（明確標注「未驗證」）| ⚠️ 中，保留但標注 |
| 技術護城河推斷 | ❌ 移除，純 AI 幻覺風險過高 | — |

**備忘錄定位**：「事實層扎實、推斷層誠實標注」的 PE 初篩備忘錄，每一筆資訊都有原始依據。

## 五、多格式文件支援

| 格式 | 處理方式 | 可靠度 |
|------|----------|--------|
| PDF | Gemini 原生讀取 | 高 |
| Word (.docx) | 文字抽取後交 Gemini | 高 |
| Excel (.xlsx) | 轉 CSV 後結構化處理 | 中（複雜表格有誤差）|
| 掃描件 / 圖片 PDF | OCR 後再處理 | 中低（依掃描解析度）|
| 純文字 (.txt) | 直接處理 | 高 |

**處理失敗行為**：系統在公司 profile 頁產生警示紀錄，標注「XX 文件未能成功解析，請手動確認」，不靜默失敗。

## 五、資料夾模式偵測

系統以文件所在資料夾判斷萃取模式：

| 資料夾 | 萃取模式 | 說明 |
|--------|----------|------|
| `raw_inputs/` | 選擇性萃取 | 外部文件，找出對目標公司的片段提及 |
| `companies/X精密/documents/` | 結構化萃取 | 公司專屬文件，對全文做完整 Insight 萃取 |

**文件誤放偵測**：文件進入 `documents/` 時，系統自動確認相關性。比例偏低時跳出警示建議移至 `raw_inputs/`，不強制阻擋。

## 六、技術架構

```
python librarian.py --new-track "被動元件"
      ↓
Gemini 即時生成 Layer 2/3 推薦公司清單（JSON）
      ↓
生成研究簡報 → 寫入 Obsidian Vault（3 秒內完成）
      ↓
背景執行軌道三爬取（TIIP / 政府採購網 / SBIR）
      ↓
爬取結果 Append 至研究簡報

━━━ 分析師上傳 PDF 後 ━━━

raw_inputs/ 資料夾
      ↓
librarian.py（Watchdog 自動監控）
      ↓
source_type 判斷 → 對應萃取 Prompt
      ↓
Gemini 2.5 Flash API（原生 PDF 讀取，支援中文字型）
      ↓
Pydantic 強制 JSON 格式校驗
      ↓
Append 至賽道樹狀圖 + Sweet Spot 交叉比對
      ↓
Log 檔記錄（每次處理結果）
```

### 技術棧

| 元件 | 選型 | 理由 |
|------|------|------|
| AI 模型 | Gemini 2.5 Flash | 原生 PDF 讀取、免費額度、中文無亂碼 |
| 資料夾監控 | watchdog（Python）| 自動偵測新檔案，無需手動觸發 |
| 資料校驗 | Pydantic | 強制 JSON 格式，確保 100% 可解析 |
| 知識庫介面 | Obsidian Vault | KM 功能內建、Graph View 可視化、WikiLink |
| 賽道設定 | tracks.yaml | Config 驅動，新增賽道不改程式碼 |
| 環境變數 | python-dotenv + .env | API Key 統一管理 |

## 七、公司卡片格式（Markdown）

發現的每間公司以 Markdown 表格呈現，欄位如下：

```markdown
| 公司名 | 在做什麼 | 上市狀態 | 來源軌道 | 關係類型 | 關係對象 |
|--------|---------|---------|---------|---------|---------|
| 漢辰科技 | 化合物半導體離子佈植設備 | 未上市 | TIIP | — | — |
| 國創半導體 | 半導體設計新創 | 未上市 | 軌道二 | 策略投資方 | 國巨(2327) |
| 凱美電機 | 被動元件材料 | 興櫃掛牌 | 軌道二 | 策略投資方 | 國巨(2327) |
| XX供應商 | 特殊探針卡材料 | 未上市 | 軌道一 | 採購客戶 | 京元電子(2444) |
```

**欄位說明**：
- **上市狀態**：`上市` / `上櫃`（自動排除，不進表格）/ `興櫃掛牌` / `未上市`
  - 比對來源：公開資訊觀測站 TWSE/OTC + 興櫃名單（本機快取）
- **關係類型**：`採購客戶`（軌道一）/ `策略投資方`（軌道二）/ `—`（軌道三）
- **關係對象**：提供線索的那間大廠

**手動新增範本**（複製貼至表格末行）：
```
| 公司名 | （填入在做什麼）| 未上市 | Manual | （填入關係類型）| （填入關係對象）|
```
並在表格下方補充：`*線索來源：XXX（如：Expert Network、行業會議）*`

## 八、Pydantic 資料結構

```python
from pydantic import BaseModel, Field
from typing import List, Literal, Optional

class ExtractedEntity(BaseModel):
    entity_name: str = Field(description="未上市或興櫃公司之法定全稱或明確簡稱")
    source_type: Literal[
        "Annual_Report",       # 軌道一：年報主要進貨商表
        "CSR_Report",          # 軌道一：CSR 去匿名化 / 衝突礦產名單
        "Financial_Statement", # 軌道二：財報附表九轉投資揭露
        "Gov_Grant",           # 軌道三：政府補助公告
        "Exhibition",          # 軌道三：展會特殊專區名冊
        "Job_Posting",         # 補充：104 職缺賽道信號
    ]
    evidence_quote: str = Field(description="100% 複製原文之關鍵事實文本")
    page_number: int = Field(description="年報/財報頁碼；文字檔填 0")
    context_relation: str = Field(
        description="與來源公司的物理事實關係。"
                    "軌道二請填入：持股比例、原始投資成本、期末帳面價值"
    )
    unlisted_evidence: str = Field(
        description="文件中支持此公司為未上市／興櫃的原文依據；若無明確依據禁止輸出"
    )
    confidence: Literal["high", "medium", "low"] = Field(
        description="high=文件明確無股票代號；medium=未見代號但未明確排除；low=有模糊跡象，需人工確認"
    )

class TrackTreePayload(BaseModel):
    track_name: str = Field(description="賽道代號，如 semiconductor_advanced_packaging")
    source_company: str = Field(description="提供情報的來源公司名稱，如 弘塑科技")
    discovered_entities: List[ExtractedEntity]
```

你是一個極度嚴謹的私募股權投資（PE）數據審計 Agent。

## 十三、設計原則與邊界

| 原則 | 說明 |
|------|------|
| 事實鐵紀律 | Prompt 硬性禁止 AI 推論，無原文不輸出 |
| Append-Only | 重複公司只加新佐證，不覆蓋舊資料 |
| 本機優先 | 所有資料存於本機 Obsidian Vault |
| 三軌交叉信號 | 多軌同時出現的公司自動標記高優先 |
| 架構可擴充 | 未來加 sourcing agent 只需插入 raw_inputs/ 前端 |
| 賽道可切換 | 改 tracks.yaml 即可，不動程式碼 |

### Watchdog 防護機制

| 防護層 | 機制 | 防止的問題 |
|--------|------|-----------|
| Debounce | 收到 modified event 後等 1.5 秒，靜止後才執行 | Obsidian 連續儲存觸發多次 |
| Processed Set | 維護 `processed_files.json`，已處理過的路徑直接跳過 | 重複處理同一文件 |
| 路徑檢查 | 偵測到路徑含 `02_Fact_Vault` 則不處理 | 誤觸發 Vault 內的檔案 |
